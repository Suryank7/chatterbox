MERN Chat App - Challenges & Fixes Log
======================================

Below is a comprehensive list of all the bugs, challenges, and issues we diagnosed and successfully fixed during our session:

1. React Context & Hooks Errors
   - Fixed "useEffect is not defined" and "useState is not defined" crashes in ChatContext.jsx, ChatContainer.jsx, and RightSidebar.jsx by adding the missing React imports.
   - Fixed real-time socket events not triggering. The `useEffect` hook in `ChatContext.jsx` that subscribed to new messages had an empty dependency array `[]`. It was updated to `[socket, selectedUser]` so it would properly attach to the socket when the user connected and update when switching chats.

2. Syntax & Destructuring Bugs
   - Fixed Vite build crashes caused by trailing commas and malformed JSX in `Sidebar.jsx`.
   - Fixed broken CSS strings containing typos that prevented Tailwind classes from applying.
   - Fixed a typo in `ProfilePage.jsx` where the user's name was incorrectly mapped to `fullname` instead of `fullName`.
   - Fixed the `FileReader` implementation in `ProfilePage.jsx` for handling image selection correctly.

3. Backend Validation & Controller Crashes
   - Fixed a MongoDB validation crash: `Message validation failed: senderId: Path 'senderId' is required`. The `messageController.js` was incorrectly trying to pull the sender ID from the URL parameters instead of the authenticated `req.user._id`.
   - Fixed typos in `io.to(receiverSocketId)` logic inside `messageController.js` to ensure the correct socket receives the emitted event.
   - Fixed a bug in `userController.js` where the `updatedUser` was not returning the properly updated document after profile edits.

4. Server Disconnections (Socket.io)
   - Fixed the `ERR_CONNECTION_REFUSED` client polling error. This was happening because the Node server was entirely crashing when hitting the `senderId` validation error. We resolved this by fixing the backend logic and moving to a watch-mode development script (`npm run server`) so the server stays resilient.

5. DOM and Accessibility Warnings
   - Added appropriate `autocomplete` tags (e.g., "current-password") to authentication forms to resolve browser security and accessibility warnings.

6. Cloudinary 403 Forbidden Upload Error
   - Diagnosed an issue where image uploads failed with "Server returned unexpected status code - 403".
   - Discovered this was due to Cloudinary API Key permission restrictions (the key lacked "create" access).
   - Added an automatic fallback mechanism in both `messageController.js` and `userController.js`: if Cloudinary throws an error or rejects the upload, the server safely intercepts it and saves the raw base64 image data directly into MongoDB instead, ensuring images always send even if third-party services fail.

7. Missing Database Schema Fields (The "Small Mark" Bug)
   - Diagnosed an issue where users saw a small empty bubble instead of the actual image they sent.
   - Discovered the `image` field was entirely missing from the `server/models/Message.js` Mongoose schema. Because of this, Mongoose was quietly stripping out all image data before saving the message to the database.
   - Added `image: { type: String }` to the `Message` model schema, fully resolving the broken image rendering issue.
